這篇是 InnerSoul 實戰系列的第 4 篇。

前面三篇已經先處理了播放器資料流、Diary 資料流，以及 UI 狀態設計。到了這一步，我開始把商業規則真正接進來，處理會員方案如何影響播放與日記功能。

## 問題背景

在 InnerSoul 裡，會員限制不是單一頁面的條件判斷，而是會同時影響不同功能。

例如：

- 播放器有試聽限制
- Diary 有可查看或可寫入的限制
- 某些內容只有特定方案能使用

一開始如果把這些規則直接寫在各個 component 裡，短期看起來很快，但很快就會出現問題：

- 播放頁跟日記頁的限制邏輯不一致
- 同一個會員方案在不同地方判斷不同
- UI 看起來有擋，但實際資料流沒真的限制住

這些問題本質上不是畫面問題，而是 **權限規則沒有被集中設計**。

## 核心思考：先定義誰可以做什麼

我後來開始先不寫畫面，而是先把需求拆成「權限規則」。

例如會員系統真正要回答的是：

- 這個使用者現在是 free 還是 pro
- 這個功能是可見、可讀、可操作，還是完全不可用
- 這個限制是作用在播放、日記，還是整個功能入口

我會先整理出這種結構：

```js
const permission = {
  canPlayFullTrack: plan === "pro",
  canReadDiary: plan === "pro" || isPastRecord,
  canWriteDiary: plan === "pro" || isToday,
};
```

這樣做的目的不是把邏輯寫死，而是先讓規則有明確的名字。

當規則有名字之後，畫面、事件與 API 才會知道該怎麼配合。

---

## 播放限制不是按鈕問題，而是權限問題

在播放器裡，一個很實際的需求是：免費會員只能試聽部分內容。

如果只是把按鈕 disabled，看起來像是做完了，但其實不夠。

因為真正要限制的是「播放行為」，不是按鈕樣式。

我後來會先把判斷抽成獨立邏輯：

```js
const canPlayIndex = index => {
  const plan = authStore.getUserPlan();
  return plan === "pro" || index < 3;
};
```

接著在真正觸發播放之前先檢查，而不是等 UI 顯示完才補救。

這件事很重要，因為：

- 權限判斷集中管理
- UI 只負責表達狀態
- 播放行為不會因為不同按鈕而出現漏網情況

## Diary 限制更像狀態切換

Diary 這個功能更明顯，因為它不是單純的可用或不可用，而是會有多種情境：

- 今天可以寫
- 過去可以看
- 未來日期不能寫
- 免費會員只能看部分內容

所以這裡我不會只寫一個 `isPro` 就結束，而是拆成多個狀態：

```js
const hasDiary = !!diary;
const isFutureSelected = selectedKey > todayKey;
const canUseDiary = plan === "pro";
const showSubscribeNotice = !canUseDiary && !hasDiary;
```

這樣 UI 在 render 時就會很清楚：

```js
if (isFutureSelected) return <DiaryWriteBlocked />;
if (showSubscribeNotice) return <SubscribeNotice />;
if (hasDiary) return <DiaryContent />;

return <EmptyDiaryState />;
```

這裡的重點不是 JSX，而是我把「權限限制」和「畫面狀態」接在一起了。

---

## 不要讓每個頁面自己發明規則

這類功能最怕的不是邏輯多，而是每個頁面都自己判斷一次。

例如：

- 播放器自己寫一套 `isPro`
- Diary 自己再寫另一套 `isFree`
- 訂閱頁又有第三套命名

最後很容易出現一個情況：

同樣是 free 會員，在 A 頁被擋，在 B 頁卻能用。

所以我後來的做法會比較偏向：

- 會員身份集中取得
- 權限邏輯集中命名
- UI 只吃已經整理好的 permission state

這樣不只維護起來比較穩，也比較不容易出現規則分裂。

---

## 我在這個功能學到的事

這段實作讓我很明顯意識到，前端不只是「把限制做出來」，而是要思考：

**限制應該被放在哪一層。**

如果限制只存在於 UI，那通常不夠穩。

如果限制能夠進入資料流與事件流程，整體系統才會一致。

這也是我在做 InnerSoul 時一個很大的轉變：

- 不再只是寫畫面條件
- 開始設計功能權限的資料流
- 讓播放、日記、會員方案之間的規則能夠對得起來

---

## 結語

訂閱會員限制這件事，看起來像是商業規則，但實際上很考驗前端的系統設計。

因為它不只會影響某一個按鈕，而是會影響整個功能流程。

透過 InnerSoul 的播放與 Diary 功能，我慢慢確定一件事：

真正穩定的前端，不只是把資料顯示出來，而是能把會員權限、功能限制與 UI 狀態整合成同一套資料流。

## 系列收束

把會員權限也接進來之後，這組系列就形成了完整的路徑：

播放器資料流 → Diary 資料流 → UI 狀態設計 → 會員權限整合

對我來說，這也是 InnerSoul 這段實作最有價值的收穫。前端不只是把功能做出來，而是把資料流、畫面狀態與商業規則接成一套能維護的系統。
